# 認証・アカウント統合／店舗管理者モデル 再設計（設計ドキュメント）

作成日: 2026-07-02
対象: 認証（サインアップ／ログイン）、アカウント統合、店舗＝組織＋管理者モデル、招待、owner ライフサイクル、モード切替 UX

このドキュメントは実装前の合意事項をまとめた契約書。generator はこの要件を全て満たすこと。UI 詳細は designer / 既存トークンに従うが、機能・データ・権限・フローはこの通りに実装する。

---

## 0. 背景と方針

- 現状はパスワードレス（メールのマジックリンク＋Google）で、店員ログインと店舗ログインが別画面（`/staff/login`・`/store/login`）。店舗は `store.owner_auth_user_id` の「1オーナー共有」。
- これを **「1人1ログインの統合アカウント」＋「店舗＝組織に個人管理者がひも付く」** モデルへ再設計する。
- **重要な前提（変わらない）**: 店舗はお金を持たない。投げ銭は Direct charge で各店員自身の Stripe 連結口座へ直接入金。よって店の削除・owner 交代は資金に一切影響しない（管理上の話）。

## 1. アカウント／認証

### 1.1 ログイン手段（全員共通・人単位）
- **メール＋パスワード**（新規追加）
- **Google OAuth**（継続）
- **マジックリンク（signInWithOtp）は廃止**（コードから除去・UI から除去）
- ログイン手段は「人」に紐づく。店員用／店舗用で分けない（統合アカウントのため）。

### 1.2 サインアップ
- メール＋パスワードで新規登録できる。
- **登録時メール確認あり**（Supabase の email confirmation）。確認前はログイン不可。
- **パスワード強度**: 最低8文字以上。弱すぎるもの（極端に短い等）は弾く（フロント＋Supabaseのポリシー）。
- **パスワード再設定**: メール経由のリセット（`resetPasswordForEmail` → 再設定ページ）を用意。

### 1.3 ログイン入口は統合
- **店員用/店舗用の別ログイン画面を廃止**し、**単一のログイン／サインアップ画面**に統合する。
- `/staff/login`・`/store/login` は統合ページへ集約（または統合ページにリダイレクト）。
- ログイン後の遷移は「1.5 初回フロー」「4. モード切替」に従う。

### 1.4 統合アカウントの構造
- **1人＝1ログイン（Supabase auth.users の1行）**。
- そこに次の役割／attachment が付く（兼任可）:
  - **店員プロフィール**（`staff` 1行。表示名・一言・avatar。チップ受取の表示にも流用）
  - **店舗の管理者**（`store_admin` 行。role=owner|admin）
- **1人＝1プロフィール**（`staff`）。店を管理するだけの人もこのプロフィールを持つ。

### 1.5 初回フロー（サインアップ後）
1. アカウント作成（メール＋パスワード or Google）
2. （メール＋パスワードの場合）**メール確認** → 確認後ログイン可
3. **プロフィール作成**（表示名・一言〔任意〕）… **全員が最初に必ず通る**（招待の有無に関わらず）
4. **店員ホームに着地**（デフォルト）
   - 招待リンク経由なら → その店に自動参加した状態で着地
   - 招待なしなら → 「招待から店に参加」「店を開設する」導線のある空のホーム

## 2. データモデル変更

### 2.1 新規: `store_admin`（店舗の管理者＝人×店＋ロール）
| カラム | 型 | 備考 |
|---|---|---|
| id | uuid PK | |
| store_id | uuid FK → store.id | |
| auth_user_id | uuid | Supabase auth.users（FK なし・文字列保持。既存 `store.owner_auth_user_id` と同方式） |
| role | text | `owner` \| `admin` |
| created_at | timestamptz default now() | 昇格順の判定（自動継承の「最古参」）に使う |
| left_at | timestamptz null | **論理削除**（管理者を外した/抜けた）。null=在籍中 |

- 一意制約: `(store_id, auth_user_id)`（同一人が同一店で二重管理者にならない）。
- **owner は各店に常に1人**（`role='owner' AND left_at IS NULL` が店ごと1行）。DB では部分ユニークインデックス等で担保、かつアプリロジックで不変条件を守る。

### 2.2 `store` の変更
- `owner_auth_user_id` は **`store_admin`（role=owner）へ移行**。移行後、store 側の owner カラムは撤去（または非使用化）。
- **論理削除カラム追加**: `closed_at timestamptz null`（null=営業中／値あり=閉店＝論理削除）。物理削除しない（受取履歴が store_id を参照するため）。

### 2.3 招待 `staff_invite` の拡張（招待の種類）
- **種類フラグ追加**: `type text not null default 'staff'`（`staff` \| `admin`）。
- **管理者招待の受け手を保持**: `accepted_auth_user_id uuid null` を追加（admin 招待の消費者は staff とは限らないため）。既存 `accepted_staff_id` は staff 招待用に維持。
- 発行・受け入れの仕組みは共通（コード＋リンク＋accept）。受け入れ時に type で分岐:
  - `staff` → `staff_store` 行（在籍・QR発行）
  - `admin` → `store_admin` 行（role=admin）
- 既存データは全削除済みのため移行不要。スキーマ変更のみ。

### 2.4 マイグレーション
- Drizzle スキーマ（`packages/db/src/schema/`）を更新し、`pnpm db:generate` → `pnpm db:migrate` で実 DB に適用（最新は 0011。続番で追加）。

## 3. 権限モデル

### 3.1 ロールと権限
| 操作 | owner | 管理者(admin) |
|---|---|---|
| 店情報の編集（名前・ロゴ等） | ✅ | ✅ |
| スタッフ招待の発行・取消 | ✅ | ✅ |
| 記録（感謝・件数）の閲覧 | ✅ | ✅ |
| 在籍スタッフの管理（在籍解除など） | ✅ | ✅ |
| **管理者の招待・削除** | ✅ | ❌ |
| **owner の譲渡** | ✅ | ❌ |
| **店の削除（閉店）** | ✅ | ❌ |

- ＝**スタッフ招待は owner＋管理者**、**管理者招待・owner譲渡・店削除は owner のみ**。
- 店舗系 API の権限チェックを、現状の「`owner_auth_user_id` 本人か」から **「その店の active な `store_admin` か（必要に応じ role=owner か）」** に置き換える（`requireOwnedStore` 相当を `requireStoreAdmin` / `requireStoreOwner` に再編）。金額は引き続き店スコープに出さない。

### 3.2 招待の発行権限
- **スタッフ招待（type=staff）**: owner＋管理者が発行可。
- **管理者招待（type=admin）**: owner のみ発行可。
- ロールは招待時に確定（受け手はロールを選ばない）。管理者招待で付与されるのは常に role=admin。

## 4. モード切替 UX

- **全員デフォルト＝店員ホーム**。受け取る/受け取らないで分岐しない（後でチップを受け取っても画面が変わらない一貫性のため）。
- **店の管理は「別モード」**。店員ホームから切り替えて入る。
  - **兼任者（active な `store_admin` を持つ人）のみ**、ヘッダー/メニューに「店員モード ⇄ 店の管理」の切替導線を出す。
  - 店の管理者でない人には切替 UI を出さない。
- **「店を開設する」** は店員側（設定 or ホーム）に置く。押すと店を作成 → 作成者が owner（`store_admin` role=owner）→ 店の管理モードへ。
- URL の器（`/staff` と `/store`）は分けたまま。切替 UI は兼任時のみ表示。
- **判定不要の原則**: ログイン時に「店員か店舗か」を判定しない。全員 `/staff`（店員ホーム）に着地し、店管理は role の有無で切替導線を出すだけ。

## 5. owner ライフサイクル

### 5.1 原則
- 各店には常に owner が1人いる（owner 不在の店を作らない）。owner は「ただ抜ける」ことはできず、抜けるなら引き継ぐ。

### 5.2 owner 譲渡
- owner は既存の active な管理者を1人指名して owner を譲渡できる。
- 実装: トランザクションで 現 owner.role → `admin`、対象 admin.role → `owner`（＋不変条件: owner は1人）。

### 5.3 owner が辞めたい
- **他に管理者がいる** → 譲渡してから抜ける。
- **他に管理者がいない** → 「先に管理者を1人招待して譲渡」 or 「店を削除（閉店）」の二択。

### 5.4 owner が消える（③・引き継ぎ操作なしでアカウント削除等）
- **残っている active な管理者がいる** → **最古参（created_at 最小）の管理者を自動的に owner へ昇格**（店を生かす）。
- **管理者が誰もいない** → **店を削除（論理削除＝closed_at セット）**。
- ＝「管理者が残っていれば自動昇格／いなければ削除」の出し分け。

## 6. 店の削除（論理削除）と資金

- **店削除＝論理削除**（`store.closed_at` セット）。物理削除しない。
- 効果:
  - そのお店の **QR・所属（staff_store）は無効化**（新規チップ停止）。
  - **過去の受取記録・収益は保全**。店員は受取履歴フィルタで過去分を引き続き確認できる（既存の脱退と同じ扱い）。
  - **資金は全額スタッフのもの**（各自の Stripe 連結口座）。店削除・owner 交代で決済取消・返金は一切発生しない。

## 8. Supabase 側の前提（コードでは変えられない・要手動設定）

## 10. 受け入れ基準（全体）

- 1人1ログイン（メール＋パスワード／Google）、メール確認・パスワード再設定が機能。マジックリンク廃止。ログイン入口統合。
- 全員デフォルト店員ホーム。兼任者のみ店管理へ切替可能。「店を開設」で owner になれる。
- store_admin による権限制御（staff招待=owner+admin、管理者招待/owner譲渡/店削除=ownerのみ）。他店・非管理者は 403/404。
- owner 譲渡・自動継承（残管理者は最古参昇格／不在は論理削除）が仕様通り。
- 店削除は論理削除で履歴・資金を保全（資金は各スタッフの Stripe 口座で無影響）。
- 金額は本人スコープのみ（店に出さない）。秘匿値を出力しない。既存（決済/送金/料率/記録/脱退/多対多/画像/埋め込み本人確認）を非破壊。

## 11. フェーズ4（owner/管理者は店員も兼ねる＋スタッフ画面タブ再編＋複数店舗＋中央ナビ切替）

フェーズ1〜3の後の追加設計。ユーザー確認済み。フェーズ4a（兼任＋スタッフ画面再編）／4b（複数店舗＋中央ナビ切替）の2段で実装する。

### 11.1 owner/管理者は「その店の店員」も兼ねる（A）
- **店を管理する人（owner/admin）は、その店の店員（`staff_store`）でもある**＝チップを受け取れQRを持つ。
- **店作成時**：`store_admin`(role=owner) に加え **`staff_store` も作成**（作成者を店員として参加させる）。
- **管理者招待の受け入れ時**：`store_admin`(role=admin) に加え **`staff_store` も作成**。
- 既に店員として在籍中の人が管理者招待を受けたら、`staff_store` は既存を流用（重複作らず）、`store_admin`(admin) を足すだけ。
- **管理者を外す** ＝ `store_admin` を論理削除するのみ。**`staff_store`（店員）は残す**（QR・受取を維持）。
- 在籍スタッフ一覧に owner/admin も出る（店員でもあるため）。
- ※全ユーザーはプロフィール（`staff`）を持つ前提（フェーズ1）なので `staff_store` を作れる。

### 11.2 スタッフ画面のタブ再編（A-1）
- `StoreStaffPage` を **3タブ**に：**在籍中／招待中／管理者**。
  - **在籍中**：その店の全 active スタッフ（owner/admin 含む）。
  - **招待中**：**スタッフ招待＋管理者招待**の両方（種類が分かる表示）。
  - **管理者**：owner＋admin を表示。**owner は「オーナー」バッジ**。
- **招待発行を統合**：招待作成で **「スタッフとして／管理者として」を選択**（同じ導線）。発行権限は §3.2 のまま（管理者招待は owner のみ）。
- **旧・管理者画面（`/store/admins`・StoreAdminsPage）は削除**。管理者招待の別画面も統合。

### 11.3 スタッフ詳細画面での管理者操作（A-2）
- スタッフ/管理者をタップ → スタッフ詳細画面へ。そこに：
  - 既存「**スタッフを外す**」（`staff_store` 論理削除）。
  - その下に「**管理者権限を外す**」：**owner のみ**・対象が管理者(admin)のとき表示。`store_admin` 論理削除のみ（店員は残す）。
  - 「**このユーザーをオーナーにする**」：**owner のみ**・対象が管理者(admin)のとき表示＝明示的 owner 譲渡（既存 transfer-owner）。
  - **owner 自身**には「管理者権限を外す」を出さない（owner を空にできない）。代わりに「オーナーを譲渡」を促す。
- **自動継承**（owner が譲渡せず抜けた/消えた時）：最古参の**管理者**を owner へ昇格／管理者ゼロなら店を閉じる（既存フェーズ2ロジックのまま）。

### 11.4 複数店舗＋中央ナビ切替（B）
- **設定からのモード切替は廃止**。
- **店員ホームに「店舗作成」項目**を追加（既に店を管理していても常に表示＝何店でも作れる）。
- **店作成完了 → 店員ホームへリダイレクト** ＋ **ボトムナビ中央に「店舗管理⇄店員」切替ボタン**を追加。
- **初回店舗作成時のみ**、中央ボタンで切り替えられる旨の**チュートリアル（1回きりのコーチマーク）**を表示。
- 中央の切替ボタンは **店員・店舗どちらのモードでも中央に常時表示**。
- **表示条件**：管理する店を**1つ以上持つ人だけ**中央ボタンを出す（純店員には出さない）。中央は一段目立つボタン。
- 切替押下時：
  - 管理店舗が**1件** → そのまま店舗管理画面へ。
  - **複数件** → **管理店舗一覧**を表示して選ばせる（選んだ店の管理画面へ）。
- これは **owner／管理者どちらも同様**。
- バックエンド：**自分が管理する店の一覧**を返すエンドポイントを追加。店舗系画面は「選択中の店（storeId）」で解決（単一 `/store/me` 前提から複数対応へ）。金額は店に出さない。

### 11.5 フェーズ4の受け入れ基準
- owner/管理者が店作成・管理者招待受け入れで **`staff_store` も作られ**、自分の店でチップを受け取れQRを持つ。管理者を外しても店員として残る。
- スタッフ画面が 在籍中／招待中／管理者 の3タブで、招待は種別選択で統合発行、旧管理者画面は削除。
- スタッフ詳細で「管理者権限を外す」「オーナーにする」が owner のみ・対象admin時に機能（owner を空にしない）。
- ホームの「店舗作成」→ 作成 → 店員ホーム＋中央切替ボタン、初回のみチュートリアル。中央ボタンは管理店ありのユーザーのみ。切替は1件→直行／複数→一覧。owner/admin 同様。
- 金額は本人スコープのみ。秘匿値を出力しない。既存（フェーズ1〜3含む）を非破壊。

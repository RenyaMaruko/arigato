# （仮称）Arigato 要件定義書

リアル投げ銭サービス / Draft v1.0

> 接客現場で、お客さまが気に入った店員さん個人に、QRコードからその場で少額の「ありがとう（投げ銭）」を、メッセージとともに送れるサービス。

---

## 1. このサービスは何か

- お客さまが、良い接客をしてくれた**店員さん個人**に、スマホで少額のチップを送れる。
- 義務的なチップではなく、**「ありがとう」の気持ちの表現**として設計する。
- お金は店を通さず、**店員さん個人に直接**届く。ただし利用は**店が承認**している前提。

### 大事にする考え方（設計のブレない軸）

| 軸 | 中身 |
|---|---|
| 感謝 ＞ 金額 | 金額ではなく「ありがとう」を主役にする |
| 個人に紐づく | 店への支払いではなく、特定の店員さんへの承認にする |
| 金額は本人だけ | 受取額は本人のみ。他のスタッフや店長には見せない |
| キャッシュレス完結 | 現金手渡しの気まずさを消す |
| データは貯める | いつ・どの店で・誰が・どんな文脈で感謝されたかを構造化して蓄積 |
| 体験を登録の前に | 本人確認や口座登録という「重い手続き」を、価値を体験する前に求めない。先に投げ銭を受け取れて、本人確認は後から、が原則 |

---

## 2. 登場人物（ロール）

- **お客さま** … 投げ銭を送る人。アプリDL不要で使えるのが理想（QR→ブラウザ）。
- **店員さん** … 投げ銭を受け取る人。本サービスの主役、最優先で大切にする。
- **店** … 利用を承認する。お金には一切触れない。
- **運営（自分）** … サービスを提供・管理する。1人運営を想定。

---

## 2.5 対応端末・画面方針（プロジェクト共通ルール）

Web アプリとして作るが、**現場で使う3者（お客さま・店員さん・店）の画面はスマホサイズのみ**を対象とする。myfans / note のように、「Web だがスマホアプリのように見える」作り方にする。PC で開いても画面いっぱいには広げず、**モバイル幅で中央に1カラム固定**して表示する。

| ロール | 対応端末 | 理由 |
|---|---|---|
| お客さま | **スマホのみ** | QR はスマホで読む。PC で開く動線が存在しない |
| 店員さん | **スマホのみ** | 接客の合間に自分のスマホで確認・操作する |
| 店 | **スマホのみ** | 店長が現場で使う。承認・スタッフ追加・感謝閲覧はスマホで完結する軽い操作 |
| 運営（自分） | **PC** | 1人運営の管理ダッシュボード。一覧・集計・取引管理はデスクトップが効率的 |

- 現場の3者は**共通のスマホ枠の作法**で作る（PC レスポンシブ対応に労力を割かない）。
- PC デザインを作るのは**運営の管理画面（admin）だけ**。
- 将来チェーン店・本部向けに店向け PC 版が必要になったら追加する（現時点ではスコープ外。8章・B2B 温存方針と一致）。

## 3. 機能要件

### 3.1 お客さま向け

- [ ] QRコードを読み取ると、投げ銭画面が開く（できればアプリ不要）
- [ ] 送り先の店員さんの名前・一言が表示される
- [ ] ¥100 / ¥300 / ¥500 などの**定額ボタン**で金額を選ぶ（悩ませない）
- [ ] 一言メッセージを添えられる
- [ ] 決済して送信（既存の決済手段を利用）
- [ ] 送信後に「ありがとうを届けました」の完了表示
- [ ] 多言語対応は**初期スコープ外**。まずは**日本人ユーザー**をターゲットに、日本語のみで開始する（インバウンド対応は将来の拡張。i18n の構造だけは技術スタックで担保しておく）

### 3.2 店員さん向け

- [ ] アカウント作成（メール / Google）。**口座・本人確認なしで、まずQR発行まで到達できる**
- [ ] 店からの**招待リンク/コード**で、所属する店に紐づいてアカウント作成する（後述 3.3 のスタッフ追加方式A）
- [ ] **自分の個人QRを発行**（名札・エプロン用に印刷できる形）。本人確認前でも発行・受付できる
- [ ] QRは `/tip/店員ID` を指す**固定URL**（一度発行したら変わらない。再発行・失効は当面なし）
- [ ] 本人確認・受取口座の登録は**後回しでよい**。未登録の間も投げ銭は「届いた感謝」として溜まり、本人確認の完了後にまとめて着金する（後述の「体験を登録の前に」）
- [ ] 投げ銭が届いたら通知（「〇〇さんからありがとうが届きました」）
- [ ] 受取履歴を見られる（**金額は本人のみ**）
- [ ] お客さまからのメッセージを読める
- [ ] **確定申告に使えるデータ**（受取額の記録）を出力できる

### 3.3 店向け

- [ ] 導入の**承認**ができる（就業規則上の整合を取るための一手間だけ）
- [ ] 所属スタッフの一覧管理
- [ ] **スタッフ追加は方式A（店が招待 → 店員が参加）**：店が招待リンク/コードを発行し、店員はそれ経由でアカウント作成して所属が確定する（店承認が招待で自然に担保される）。店員のQR自体は**店員本人**が発行する（店は発行主体ではない）
- [ ] 感謝の可視化：**「ありがとうの件数」「お客さまの声」**が見える
  - 店全体の件数（今日/今週/今月）
  - お客さまの声（メッセージ・いつ・誰宛か）を時系列フィードで表示
  - スタッフ別の件数も見せてよいが、**順位付け・件数順の並べ替えはしない**（名簿順など中立な並びにして序列を作らない）
- [ ] ❌ 金額・金額ランキングは**見せない**（職場の序列を持ち込まない）
- [ ] ❌ 店宛のチップは存在しない（投げ銭は店員さん個人にのみ。店はお金に触れない）

### 3.4 運営向け

- [ ] 店・店員アカウントの管理
- [ ] 感謝データの蓄積（構造化して保存）
- [ ] 取引・手数料の集計

---

### 3.5 設計思想：受け取りの摩擦は「後ろ倒し」にする

投げ銭を**受け取る**には、Stripe Connect を通じた本人確認（犯収法対応・身分証・銀行口座登録）が必須になる。これは規制上どうしても避けられない手続きだが、これを**価値を体験する前に**店員さんへ求めると、「気軽なありがとう」という世界観とギャップが生まれ、登録のハードルで脱落する。

そこで、手続きの順序を逆にする：

```
【先に】 アカウント作成 → QR発行 → 投げ銭が届く → 「ありがとう」が溜まる（＝価値を先に体験）
【後で】 一定額が溜まる / 着金したいタイミングで → 本人確認・口座登録 → まとめて着金
```

- お客さま側は最初から登録不要のまま（QR→決済のみ）。
- 店員さん側は「**受け取れる体験**」を先に味わってから、現実的な手続き（本人確認）に進む。
- これはメルカリ等の「売上は先に立つ、振込申請・本人確認は後」と同じ思想。**未確定の受取残高を安全に保持し、本人確認の完了をトリガーに着金させる**のが実装上の肝。

> 実装観点：本人確認前の受取分は「保留残高（未着金）」として記録し、Stripe Connect オンボーディング完了の Webhook をトリガーに payout 可能状態へ遷移させる。残高の保持・着金条件は Service / Model 層で明示的に管理する。

## 5. お金の流れ

```
お客さま ──（投げ銭 + 上乗せ手数料）──▶ 決済事業者
                                          │
                          ┌───────────────┴───────────────┐
                          ▼                               ▼
                  店員さんへ満額が届く              運営の取り分（手数料）

  ※ 店はお金に一切触れない（レジ・売上・経理を通さない）
  ※ 送金インフラは自前で持たず、既存の決済事業者の上に乗る
```

- 上乗せ手数料は**お客さま側**に乗せる（店員さんからは引かない＝満額が届く）。
- 収益は当面この**決済上乗せ一本**。シンプルに開始する。

## 6. 非機能要件・前提

- **低コスト運用**：1人・月数万円規模で回せる構成にする（サーバー・DB・通知など）。
- **決済は外部依存**：自前で送金しない。資金移動規制を避けるため既存決済の上に乗る。
- **プライバシー**：金額は本人のみ開示。これは原則として徹底する（後から店長向け金額レポート等を足さない）。
- **データ蓄積の質**：単なるテキスト保存でなく、いつ・どの店・誰・どんな文脈、を構造化して残す（将来の選択肢のため）。

---

## 7. 今回作らないもの（スコープ外）

- ❌ 店向けの有料ダッシュボード（B2Bサブスク）→ **将来の選択肢として温存**。今は作らない。
- ❌ 自前の送金・決済インフラ → 既存事業者に任せる。
- ❌ 金額ランキング、店長向けの金額レポート → 原則に反するので作らない。

---

## 8. 制約・気をつけること

- **資金移動 / 決済規制**：自前送金は規制が重い。既存決済の上に乗る前提を崩さない。
- **就業規則 / 労務**：店に無断利用は店員さんに違反リスク。だから**店承認を前提**にして店員さんを守る。
- **税務**：継続受取は申告対象になりうる。記録データを本人に提供して、後から困らせない。

> 法務・労務・税務の最終判断は、弁護士・社労士・税理士に確認する。

---

## 9. まだ決まっていない / 検証したいこと

- **採用率**（1店あたり1日に何件投げ銭が出るか）＝ 事業の生命線。実証で確かめる。
- お客さま画面で「他の人の投げ銭額」を見せるか問題（功罪あり、未決）。
- サービス名（「Arigato」は仮）。
- 店メリットが「本当に定着に効くのか」＝ 将来B2Bに進むかの判断材料。

---

## 10. まず最初にやること（最小の検証）

1. ツテのある**小規模な店を1〜2軒**見つける（まずは日本人客が中心の店）
2. 「お金は店を通しません、置かせてください」と**店公認**で依頼
3. 店員さんの個人QR + 卓上案内（日本語）を置く
4. **お客さまが本当にその場で感謝を送るか**を観察する

→ ここがすべての出発点。最小で確かめてから広げる。
